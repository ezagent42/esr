# PR-20 Spec: Session cwd + tag/worktree redesign

**Status:** brainstorm — answering grill questions before locking design.
**Author:** allen (linyilun) + claude pair-prog session.
**Date:** 2026-04-28.
**Implementation PR:** PR-20 (after this spec is locked).

## Background

Today's mental model (v0.2):

```
adapters.yaml:        "esr_dev_helper" → {feishu app credentials}
workspaces.yaml:      "esr-dev"         → {cwd, start_cmd, role, chats}
/new-session esr-dev tag=root  → spawns a session bound to "esr-dev",
                                  cwd = workspaces["esr-dev"].cwd
/new-session esr-dev tag=child → spawns a SECOND session, ALSO cwd =
                                  workspaces["esr-dev"].cwd
                                  ↑ both CCs editing same files →
                                    user worry: "lock-step conflicts"
```

The user's pain (verbatim from grill, 2026-04-28 08:09):

> 我不希望 new-session 全部在同一个 workspace 中，这样容易出现同时修改代码的情况。我希望每次 new-session 都要显式指定工作目录（cwd），tag 应该改名叫 worktree，根据 tag 的命名，创建新的 worktree 作为 cwd 的分支。

Translation: each `/new-session` should isolate to its own git
worktree; `tag` is doing two jobs (session label + workspace
sub-key) and should be renamed `worktree` to make the intent
explicit. New worktrees are auto-created from the worktree name.

Cousin observation: the term "tag" is also currently overloaded with
"the substring after `@` in `@tag-name <body>` for routing" (per
`docs/cookbook.md` §"Route a message to a specific session"). That
routing prefix is unrelated to the `/new-session` label and stays
intact in this spec.

## Proposed direction (sketch)

Two-axis model:

```
workspace (template / persistent config)
├── name: esr-dev                          ← addressed by chat
├── chats: [{chat_id: oc_…, app_id: …}]    ← which IM chats route to it
├── role: dev                              ← CC's CLAUDE.md prelude
├── start_cmd: scripts/esr-cc.sh           ← how to spawn CC
├── default_branch: main                   ← NEW: new worktrees fork from here
├── worktrees_root: ~/Workspace/esr/.claude/worktrees/  ← NEW (or default)
└── (NO MORE --cwd at workspace level)

session (concrete instance)
├── workspace: esr-dev
├── worktree: feature-foo                  ← was `tag=feature-foo`
├── cwd: <worktrees_root>/feature-foo      ← derived; auto-created on spawn
└── branch: feature/foo (or autogenerated) ← NEW: bound to git branch
```

`/new-session esr-dev worktree=feature-foo` then:
1. Resolves cwd = `${workspace.worktrees_root}/feature-foo`.
2. If that path doesn't exist: `git worktree add <cwd> <new branch from default_branch>`.
3. Spawns CC with that cwd.

## Grill questions (answer these to lock the design)

The following 7 questions cover the open mechanics. **Each can be
answered in 1-3 sentences**; "I don't care, pick a default" is a
valid answer when neither option is obviously better.

---

### Q1: Old worktree on session end — keep, prune, or operator's choice?

When `/end-session` fires for a worktree-bound session, what happens to
the worktree on disk?

**(a) Always keep.** `git worktree remove` is destructive (loses
uncommitted state); operators clean up by hand later via
`git worktree remove` once they've confirmed nothing's lost.
Failure mode: disk fills up; needs periodic sweep.

**(b) Always prune.** Session end auto-runs `git worktree remove
--force <path>`. Failure mode: operator typed `/end-session` while
they had uncommitted work in the worktree → silently lost.

**(c) Prune iff clean.** Run `git status` first; remove only if
working tree is clean. If dirty, log a warning + leave it. Failure
mode: more code to maintain, edge cases around submodules / ignored
files.

**(d) Per-session flag.** `/end-session foo --remove-worktree` opts in.
Default = keep.

**Recommend (c) — prune iff clean.** Auto-cleanup handles the 95%
case of "I tried this branch, didn't pan out, end the session"
without losing the operator's in-progress work. Edge cases (submodules,
ignored files) we document and accept.

---

### Q2: Multiple sessions in the same worktree — allowed or forbidden?

`/new-session esr-dev worktree=feature-foo` twice in a row. Two
behaviors possible:

**(a) Forbidden.** Second call returns "worktree already in use by
session X". Rationale: per-worktree exclusivity is the whole point of
this redesign; two sessions in one worktree puts us back in the
"two CCs editing same files" hazard.

**(b) Allowed but shared.** Both sessions run with the same cwd; no
`git worktree add` for the second. Rationale: occasionally useful for
"run two CC instances in parallel on the same code, e.g. one
implementing, one writing tests".

**(c) Allowed but second creates a uniquified worktree.**
`feature-foo` → `feature-foo-2` automatic. Rationale: never
fail an operator command; just disambiguate.

**Recommend (a) — forbidden.** This is what the user asked for: avoid
the "two CCs in one dir" race. If they want parallel sessions, they
use distinct worktree names.

---

### Q3: Default branch for new worktrees — what does it fork from?

`git worktree add <path> <branch>` needs a branch. When operator
types `/new-session esr-dev worktree=feature-foo`:

**(a) Fork from `main`.** Always cuts a new branch named
`feature-foo` (or `worktree/feature-foo`?) starting at `origin/main`.
Predictable, reproducible.

**(b) Fork from current HEAD of the workspace's repo.** Operator's
current state determines the start point. More flexible, less
reproducible.

**(c) Workspace-configurable.** `default_branch: main` field in
`workspaces.yaml`; per-workspace override available. Defaults to
`main` if absent.

**(d) Per-session override.** `worktree=feature-foo from=develop`.
Overengineered for v1.

**Recommend (c) — workspace-configurable, default `main`.**
Predictable for the common case + escape hatch for projects on
`master` / `develop` / etc. Per-session override is a future PR.

---

### Q4: Branch name from worktree name — verbatim, prefixed, or operator-chooses?

When `git worktree add <cwd> <new_branch>` runs, what is `<new_branch>`?

**(a) Verbatim.** `worktree=feature-foo` → branch `feature-foo`.
Pro: shortest. Con: collides with branch naming conventions
(`feature/`, `fix/`, `chore/` prefixes).

**(b) Prefixed.** Always `worktree/<name>`. Pro: branches starting
with `worktree/` are obviously esrd-managed. Con: extra typing
when grepping git log.

**(c) Operator chooses.** `worktree=feature-foo branch=feature/foo-thing`.
Pro: max flexibility. Con: more args to type.

**(d) `<workspace-name>/<worktree-name>`.** e.g. `esr-dev/feature-foo`.
Pro: branches sorted by workspace in `git branch` output. Con: slash
in branch names sometimes annoys tools (`git branch -D` requires
exact name).

**Recommend (a) — verbatim.** Operator chose the worktree name; trust
them. If they want `feature/foo`, they type `worktree=feature/foo`
(slash should round-trip cleanly into both directory name and branch
name on macOS / Linux).

---

### Q5: Workspace `--cwd` field — keep for backwards compat, deprecate, or delete?

Current workspace yaml has `cwd: /path`. Post-PR-20 we want
worktrees-root + per-session cwd derivation.

**(a) Delete.** Remove the field; old `workspaces.yaml` files fail to
load until operator migrates. Loud + forces migration but disruptive.

**(b) Deprecate, treat as worktrees_root if present.** Old `cwd:`
field continues to load but logs a warning. New
`worktrees_root:` field takes precedence. `cwd:` removed in 6 months.

**(c) Repurpose.** Rename: workspace.cwd → workspace.worktrees_root in
the yaml schema. ESR shipping registers a `cwd` → `worktrees_root`
auto-rename in the loader.

**(d) Keep both, distinct semantics.** `cwd:` = "fallback for
sessions without `worktree=`"; `worktrees_root:` = "for sessions
with `worktree=`". Behavior depends on whether the slash command
named a worktree.

**Recommend (b) — deprecate gracefully.** Existing `workspaces.yaml`
files keep working; operators see a warning + migrate at their leisure.
"Delete after 6 months" is a real future-PR item, not a vague
threat — log a one-liner warning and we ship.

---

### Q6: Slash command spelling — `worktree=` vs `--worktree=` vs positional?

Today: `/new-session <ws> tag=<x> [thread_id=<y>]`. Three options for
the new term:

**(a) `worktree=<name>`.** Same syntax as today's `tag=`. Predictable
for existing users.

**(b) `--worktree=<name>`.** GNU long-option style. More CLI-ish but
doesn't match other slash command syntax.

**(c) Positional second arg.** `/new-session esr-dev feature-foo`.
Shortest. But conflicts with future args.

**Recommend (a) — `worktree=<name>`.** Mirrors existing `key=value`
syntax operators already type. Migration path: `/new-session esr-dev
tag=root` keeps working with a one-shot rename to
`worktree=root` in the parser (no behavior change since `tag` and
`worktree` mean the same thing post-PR-20).

---

### Q7: Initial worktree path layout — flat or nested per workspace?

`worktrees_root` is a directory; new worktrees go inside.

**(a) Flat under workspace root.**
`<worktrees_root>/<worktree_name>` →
`~/Workspace/esr/.claude/worktrees/feature-foo`. Simple. Conflict if
two workspaces both want `feature-foo`.

**(b) Per-workspace subdir.**
`<worktrees_root>/<workspace_name>/<worktree_name>` →
`~/Workspace/esr/.claude/worktrees/esr-dev/feature-foo`. No conflicts;
deeper paths.

**(c) Per-app subdir.** Unlikely useful; one app spans many workspaces
typically.

**Recommend (b) — per-workspace subdir.** Each workspace's worktrees
live in their own namespace. `git worktree list` already shows full
paths; the deeper depth doesn't hurt.

---

Right now (2026-04-28), the user has:
- prod esrd: 0 live sessions (no `/new-session` issued yet against
  `esr_helper`)
- dev esrd: 0 live sessions (same against `esr_dev_helper`)

So **migration risk = 0** — no in-flight sessions to break. PR-20 can
ship with breaking changes to the slash command syntax + workspace

(If the answer to Q5 is (a), skip the deprecation tail entirely.)

## Out of scope (deferred)

The following are valid future improvements but **not** part of PR-20:

- **Per-session `--cwd-override`** — a flag that bypasses the
  worktrees model for one-off "use this exact directory" sessions.
  Worth adding only after PR-20 settles in real use.
- **Worktree garbage collection job** — periodic sweep of
  `worktrees_root/*` for paths whose backing branch is gone. Operators
  can `git worktree prune` by hand for now.
- **Cross-workspace branch sharing** — checkout out the same branch
  from two workspaces. Highly speculative.

## Implementation outline (after spec lock)

Conditional on Q1-Q7 answers, expect ~3-4 distinct commits:

1. **Workspace yaml schema** — `worktrees_root`, `default_branch`
   fields; deprecation warning on `cwd:` (per Q5).
2. **`/new-session` parser** — accept `worktree=` (Q6); auto-rename
   `tag=` → `worktree=` (one-shot compat).
3. **Worktree spawn helper** — `git worktree add` + branch creation
   (Q3, Q4); per-workspace path layout (Q7).
4. **Session-end teardown** — prune-iff-clean (Q1); refuse second
   session in same worktree (Q2).

LOC budget: ~150-200 Elixir + ~50 Python (for `workspace add` CLI
schema), plus tests.

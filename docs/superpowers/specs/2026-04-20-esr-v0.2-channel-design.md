# ESR v0.2 Channel + MCP Bridge Design

Author: brainstorming session with user (linyilun)
Date: 2026-04-20
Status: draft, awaiting user review
Supersedes: nothing (additive to v0.1 which is feature-frozen at commit 50b187d)

## 1. Goal & Scope

### 1.1 In scope

1. **`esr-channel` MCP bridge** — a Python MCP stdio server that bridges
   Claude Code sessions to the ESR runtime over WebSocket, API-compatible
   (at the tool-name / payload-shape level) with cc-openclaw's
   `openclaw-channel`. One MCP process per CC session, parented by CC.

2. **Multi-app workspaces** — explicit per-chat binding so a single esrd
   can route between multiple Feishu apps. `workspaces.yaml` lists each
   chat with its owning `app_id`.

3. **Primary-path switch from tmux send-keys to MCP notification** —
   Feishu inbound messages reach the CC session as MCP-protocol
   notifications, not as simulated keystrokes in the tmux pane.

4. **Auto-restore of adapter + topology state on esrd start** — scan
   `~/.esrd/<instance>/adapters.yaml` and re-instantiate
   `feishu-app-session` for each registered feishu instance without
   requiring a manual `esr adapter add` after every restart.

5. **Final gate `--live` v2** — 6-artifact nonce correlation covering
   L1-L6 from the v0.2 acceptance checklist (§ 7).

### 1.2 Out of scope

- **L7 — group-chat multi-app routing**: v0.2 workspaces still declare
  `(chat_id, app_id)` explicitly, so the runtime knows which app a
  group's bot identity is for. A "one group, two bots, pick the right
  one based on context" resolution is deferred to v0.3.
- **Session migration / handoff**: killing a CC session and running
  `esr cmd run` to respawn is the recovery path; no in-place session
  reattachment.
- **Multiple esrd instances**: v0.2 keeps the `default` instance
  hard-coding in CLI paths; per-instance CLI awareness is v0.3.
- **Dedicated `update_title / forward / send_summary / spawn_session /
  kill_session / list_sessions` MCP tools**: all folded into
  `reply(..., edit_message_id?)` + shell-invoked `esr` CLI commands.
  Reduces the MCP tool surface from 9 to 3.

### 1.3 Non-goals (explicit anti-goals)

- **ESR v0.2 is not a replacement for cc-openclaw today.** cc-openclaw
  keeps serving its production project. v0.2 is the substrate that
  could replace it in v0.3+ if desired. Short-term coexistence is
  guaranteed by using a separate esrd instance + separate Feishu apps.

## 2. Architecture

### 2.1 Topology

```
           Feishu / Lark
                │
      ┌─────────┴─────────┐
      │                   │
      ▼                   ▼
 HTTP REST          WebSocket
 (send_message)     (im.message.receive_v1)
      │                   │
      └───────┬───────────┘
              ▼
   ┌─────────────────────────┐
   │  feishu adapter_runner  │  Python, per-app instance
   │  (esr.ipc.adapter_runner) │
   └───────────┬─────────────┘
               ▼  adapter:feishu/feishu-app:<app_id>
          Phoenix Channels (esrd, port 4001)
               ▲
   ┌───────────┴────────────────┐
   │  feishu_app_proxy PeerServer│  one per app_id
   │  /new-session → InvokeCommand│
   │  msg → Route active thread  │
   └───────────┬────────────────┘
               ▼
   ┌─────────────────────────┐
   │ feishu_thread_proxy     │  one per active CC session
   │ msg_received → notify   │
   │ cc_output → send_message│
   └───────────┬─────────────┘
               ▼  cli:channel/<session_id>
          Phoenix Channels (NEW, port 4001)
               ▲  WebSocket, per-CC-session
   ┌───────────┴──────────────┐
   │ esr-channel MCP bridge   │  Python, stdio child of CC
   │ (adapters/cc_mcp/…)      │
   └───────────┬──────────────┘
               ▼  JSON-RPC over stdio
          Claude Code process
```

### 2.2 Why this shape

- **MCP-as-sidecar** (architecture option B in the brainstorm):
  `esr-channel` is a **pure transport adapter**, not an ESR actor. It
  has no state beyond "this process belongs to session X, here is its
  WS connection to esrd." All business logic lives in handler modules;
  all lifecycle lives in PeerServer / SessionRegistry.

- **Per-session MCP process** because CC parents it via stdio; the OS
  handles lifecycle automatically (CC exits → stdio EOF → MCP exits).
  No supervisor, no pidfile, no signal handling.

- **feishu → esr-channel is the primary path**; tmux send-keys stays
  available through `cc_tmux.send_keys` but the `feishu-thread-session`
  pattern no longer relies on it for message transit. `cc_tmux` keeps
  its existing responsibilities: `new_session` (spawn CC inside tmux)
  and `emit_events` (pane polling for sentinel-prefixed lines during
  session bootstrap). Both remain useful, neither is the message path.

## 3. Components

### 3.1 New Python packages

**`adapters/cc_mcp/src/esr_cc_mcp/channel.py`** (~200 LOC)

MCP stdio server. Analogous to `cc-openclaw/channel_server/adapters/cc/
channel.py` in role, but WS peer is esrd and tool surface is narrower.

- `list_tools` — returns 3 tools: `reply`, `react`, `send_file`
- `call_tool` — serialises args into `tool_invoke` WS frame, awaits
  `tool_result` frame, returns content to CC
- Inbound `notification` WS frames → inject as MCP
  `notifications/message` so CC sees them in conversation
- Inbound `session_killed` → MCP notification with level=warning, then
  close WS, `sys.exit(0)`
- Env vars read at startup: `ESR_ESRD_URL` (default
  `ws://127.0.0.1:4001`), `ESR_SESSION_ID`, `ESR_WORKSPACE`,
  `ESR_CHAT_IDS` (JSON-encoded list)

### 3.2 New Elixir modules

**`runtime/lib/esr_web/channel_socket.ex`**

Phoenix `Socket` mounted at `/channel/socket/websocket`. No auth for
v0.2 (same as AdapterSocket / HandlerSocket). Channel route:
`channel "cli:channel/*", EsrWeb.ChannelChannel`.

**`runtime/lib/esr_web/channel_channel.ex`**

Phoenix `Channel` for `cli:channel/<session_id>`. Handlers:

- `join/3` — parse `session_id` from topic; register in
  `SessionRegistry` with `socket.transport_pid` as the WS pid; reply
  `{:ok, %{registered: true}}`
- `handle_in("envelope", %{"kind" => "session_register"}, ...)` —
  idempotent update of the registry row (chat_ids, workspace)
- `handle_in("envelope", %{"kind" => "tool_invoke"}, ...)` — look up
  `feishu_thread_proxy` PeerServer by session_id; `send(peer_pid,
  {:tool_invoke, req_id, tool, args})`; PeerServer's new handle_info
  clause translates to an `Emit` on the feishu adapter, awaits the
  adapter's `directive_ack`, and sends the tool_result back to the WS
  via `push(socket, "envelope", %{"kind" => "tool_result", ...})`
- `terminate/2` — mark session offline in SessionRegistry, broadcast
  telemetry `[:esr, :session, :offline]`

**`runtime/lib/esr/session_registry.ex`**

GenServer + ETS. State:
`%{session_id => %{peer_pid, ws_pid, chat_ids, app_ids, workspace,
status: :online | :offline, last_seen_ms}}`.

Public API:
- `register(session_id, peer_pid, ws_pid, chat_ids, app_ids, workspace)`
  — idempotent upsert, status :online
- `mark_offline(session_id)` — keep row, status :offline, set timestamp
- `lookup(session_id) :: {:ok, row} | :error`
- `list() :: [{session_id, row}]`
- `notify_session(session_id, envelope)` — look up ws_pid, push to it

### 3.3 Modified handler modules

**`handlers/feishu_app/on_msg.py`**

- Change command prefix `/new-thread` → `/new-session`
- Parse `/new-session <workspace> tag=<tag>` (workspace positional,
  `tag=` kw). Missing tag defaults to a slug of the current Feishu
  message_id.
- Add state field `active_thread_by_chat: dict[str, str]` so non-command
  messages can route to the most recent `/new-session` on the same
  chat. Unknown chat_id → drop as before.
- Add state field `last_chat_id`, captured on first msg_received, for
  future "default chat" helpers.
- Pass `{thread_id, chat_id, workspace, tag}` as InvokeCommand params.

**`handlers/feishu_thread/on_msg.py`**

- Unify on `event_type == "msg_received"` (drop the `feishu_msg_received`
  branch; fix the v0.1 naming mismatch).
- On inbound msg_received: dedup by message_id, then Emit with a
  **synthetic adapter name** `esr-channel` and action `notify_session`:
  ```python
  Emit(adapter="esr-channel",
       action="notify_session",
       args={"session_id": state.thread_id, "source": "feishu",
             "chat_id": event.args["chat_id"],
             "message_id": event.args["message_id"],
             "user": event.args.get("sender_id", ""),
             "content": event.args["content"]})
  ```
  **Important**: `esr-channel` is NOT an adapter instance in the normal
  ESR sense — no `adapter:esr-channel/*` Phoenix topic, no
  HubRegistry binding. `PeerServer.dispatch_action` special-cases
  `adapter == "esr-channel"` and routes via
  `SessionRegistry.notify_session/2` instead of the usual adapter
  broadcast (see §4.2 step 6). This keeps handlers' Emit API uniform
  while using the SessionRegistry push path underneath.
- Keep `cc_output` → `Emit(feishu, send_message)` for the scenario
  mock path; --live uses the MCP reply tool path instead.

### 3.4 New CLI verbs

**`esr workspace add / list / remove`**

```
esr workspace add <name> \
    --cwd <path> \
    --start-cmd <cmd> \
    --role <role> \
    --chat <chat_id>:<app_id>:<kind> \
    [--chat ...] \
    [--env KEY=VAL ...]
```

Writes `~/.esrd/default/workspaces.yaml` and pushes to esrd via
`cli:workspace/register` (new CliChannel dispatch) so the runtime has
it in memory.

**`esr session chat-id <session_id>`**

Looks up the workspace of a session and prints its default chat_id.
Used by CC shell commands to answer "where do I reply by default" when
a prompt doesn't include a channel tag.

### 3.5 New shell scripts

**`scripts/esr-cc.sh`**

Boot wrapper analogous to `cc-openclaw/cc-openclaw.sh`:
1. Source `~/.zprofile` / `~/.zshrc` for PATH (tmux non-interactive
   shells need this)
2. Add `/opt/homebrew/bin:/usr/local/bin:$HOME/.npm-global/bin` as
   fallback
3. Source `scripts/esr-cc.local.sh` if present (for HTTP_PROXY et al.)
4. Source `.mcp.env` for MCP server secrets
5. Parse `workspaces.yaml` for the target workspace (passed via
   `ESR_WORKSPACE`): resolve cwd, role, chats
6. Render `CLAUDE.md` template from `roles/<role>/CLAUDE.md` into the
   workspace cwd with `ESR_USER` / `ESR_SESSION` / `ESR_TAG` variables
7. Invoke claude:
   ```
   claude --permission-mode bypassPermissions \
          --dangerously-load-development-channels server:esr-channel \
          --mcp-config .mcp.json \
          --add-dir <cwd> \
          --settings roles/<role>/settings.json
   ```
8. Write `.mcp.json` at workspace cwd pointing `esr-channel` at the
   Python binary + env (ESR_ESRD_URL, ESR_SESSION_ID, ESR_WORKSPACE,
   ESR_CHAT_IDS)

### 3.6 Modified Elixir modules

**`runtime/lib/esr/application.ex`**

Startup hook `restore_state_from_disk/0` runs after supervision tree is
up. Steps:
1. Read `~/.esrd/default/adapters.yaml`
2. For each instance, ensure the adapter worker via
   `WorkerSupervisor.ensure_adapter` (uses existing idempotent path)
3. For each feishu-type instance, `Instantiator.instantiate` the
   `feishu-app-session` pattern with params `{app_id: <instance
   app_id>}`
4. Read `~/.esrd/default/workspaces.yaml` and populate an in-memory
   Workspaces.Registry so `/new-session <name>` can look up the cwd /
   start_cmd / chats

## 4. Data flow walkthroughs

### 4.1 `/new-session esr-dev tag=root` (Feishu → new CC)

1. User in Feishu DMs `ESR助手`: `/new-session esr-dev tag=root`
2. Lark WS event (or adapter's polling fallback if bot-self) → feishu
   adapter emit_events → envelope pushed on
   `adapter:feishu/feishu-app:<app_id>`
3. AdapterChannel → feishu_app PeerServer
4. feishu_app.on_msg parses command → InvokeCommand(name=
   feishu-thread-session, params={thread_id: root, chat_id: oc_...,
   workspace: esr-dev, tag: root})
5. PeerServer.dispatch_action(invoke_command) → Task.start →
   Topology.Instantiator.instantiate
6. Topology spawns thread:root / tmux:root / cc:root; tmux node's
   init_directive new_session runs `scripts/esr-cc.sh` with env
   ESR_WORKSPACE=esr-dev / ESR_SESSION_ID=root / ESR_TAG=root
7. esr-cc.sh → claude --dangerously-load-development-channels →
   esr-channel MCP starts → WS connect to esrd → session_register
8. ChannelChannel.join → SessionRegistry.register(session_id="root",
   peer_pid=<cc:root PeerServer>, ws_pid=<socket.transport_pid>, ...)
9. (Optional) feishu_thread sends an Emit(feishu, send_message) with
   "session root ready" back to the originating chat — this is the
   L1 ack artifact.

### 4.2 Inbound msg → notification to CC

1. User in same DM: `hello probe`
2. feishu adapter → adapter channel → feishu_app PeerServer
3. content doesn't start with `/` → lookup active_thread_by_chat →
   "root" → Route(target="thread:root", msg={event_type:
   msg_received, args: {content, message_id, chat_id, ...}})
4. PeerServer routes to thread:root PeerServer
5. feishu_thread.on_msg → dedup + Emit(esr-channel, notify_session,
   {session_id: root, source: feishu, chat_id, message_id, content,
   user})
6. PeerServer.dispatch_action(emit) scans HubRegistry; no
   `adapter:esr-channel/*` binding (esr-channel is NOT an adapter
   binding; it's a SessionRegistry-keyed push). New path in
   dispatch_action detects `adapter=="esr-channel"` and calls
   `SessionRegistry.notify_session(session_id, envelope)` directly
   (no adapter Phoenix channel roundtrip)
7. SessionRegistry.notify_session → `Phoenix.Channel.push(ws_pid,
   "envelope", %{"kind" => "notification", ...})`
8. MCP bridge handle_in frame → inject MCP `notifications/message`
   with the content
9. CC sees the notification in its conversation context

### 4.3 CC calls `reply` → Feishu

1. In CC, user prompt triggers tool call `reply(chat_id=oc_...,
   text="acknowledged")`
2. MCP bridge handle_call_tool → send WS `tool_invoke` frame with
   unique req_id
3. ChannelChannel.handle_in → send(peer_pid, {:tool_invoke, req_id,
   "reply", args})
4. feishu_thread PeerServer adds a new handle_info clause that
   translates tool_invoke "reply" → dispatch_action on Emit(feishu,
   send_message, args={chat_id, content: text}). Uses existing
   Emit → HubRegistry topic resolution.
5. feishu adapter_runner receives directive → Lark REST send_message
6. directive_ack flows back through PubSub `directive_ack:<id>` → peer
7. PeerServer pushes `tool_result` frame to the WS with the ack data
   (message_id)
8. MCP bridge waits on req_id → receives result → returns to CC as
   TextContent

### 4.4 `esr cmd stop` → graceful MCP shutdown

1. Operator: `esr cmd stop feishu-thread-session --param thread_id=root
   --param chat_id=oc_...`
2. CliChannel → Topology.Registry.deactivate
3. For each peer_id in handle.peer_ids (reverse topo order):
   PeerSupervisor.stop_peer
4. cc:root PeerServer.terminate fires → emits `broadcast(
   "cli:channel/root", "envelope", %{"kind" => "session_killed",
   "reason" => "manual_stop", "final" => true})` before going down
5. MCP bridge handle_in("session_killed") → stdio-inject MCP
   notification with level=warning: "ESR: session 'root' has been
   terminated by the runtime. reply/react tools will no longer work.
   Please restart CC."
6. MCP `await asyncio.sleep(0.5)` → close WS → `sys.exit(0)`
7. CC stdio EOF → CC logs "MCP server 'esr-channel' disconnected"
8. tmux window stays (tmux's remain-on-exit policy) until user closes
   it, or `cc_tmux.on_directive kill_session` in the rollback path

### 5.3 WS protocol (canonical frames)

All frames wire-encode as Phoenix v2 `["join_ref","ref","topic","event","payload"]` with `event = "envelope"` and `payload.kind` discriminator (matches v0.1 adapter/handler convention).

```
→ esrd (from MCP):
  {kind: "session_register",
   session_id: <string>,
   workspace: <string>,
   chats: [{chat_id, app_id, kind}, ...]}

  {kind: "tool_invoke",
   req_id: <uuid>,
   tool: "reply" | "react" | "send_file",
   args: <tool-specific>}

← esrd (to MCP):
  {kind: "notification",
   source: <string>,                  # "feishu" for this design
   chat_id: <string>,
   message_id: <string>,
   user: <string>,
   content: <string>,
   ts: <iso8601>}

  {kind: "tool_result",
   req_id: <uuid>,
   ok: <bool>,
   data?: <object>,
   error?: {type, message}}

  {kind: "session_killed",
   reason: <string>,
   final: <bool>}                     # always true in v0.2
```

## 6. Lifecycle contract

### 6.1 Normal CC exit

| Step | Who | What |
|------|-----|------|
| 1 | user | Ctrl+D in CC |
| 2 | CC | writes EOF to MCP's stdin |
| 3 | MCP bridge | `stdio_server()` returns → anyio task group cancels → WS `close` frame |
| 4 | ChannelChannel | `terminate/2` fires |
| 5 | SessionRegistry | `mark_offline(session_id)` — PeerServer NOT stopped yet |
| 6 | telemetry | `[:esr, :session, :offline]` event (metadata: session_id, reason=:normal) |
| 7 | timeout watchdog | after 60s offline, Topology.Registry.deactivate the parent topology (safety net — user may have intended this as a soft restart) |

### 6.2 CC crash

Same as §6.1 from step 2 onwards. The distinction (crash vs exit) lives only in the reason metadata.

### 6.3 Operator-initiated kill

| Step | Who | What |
|------|-----|------|
| 1 | operator | `esr cmd stop feishu-thread-session --param ...` |
| 2 | CliChannel | `dispatch("cli:stop/...")` → `Topology.Registry.deactivate` |
| 3 | deactivate | reverse-order `PeerSupervisor.stop_peer` on handle.peer_ids |
| 4 | cc:<sid> PeerServer | `terminate/2` runs — **before** `{:shutdown, :normal}` returns, broadcast `session_killed` to `cli:channel/<sid>` topic |
| 5 | MCP bridge | handle_in("session_killed") → stdio-inject MCP notification → sleep 500ms → close WS + `sys.exit(0)` |
| 6 | CC | stdio EOF → logs "MCP server disconnected" → CC process continues running but without the channel tools |

### 6.4 esrd restart

| Step | Who | What |
|------|-----|------|
| 1 | operator | `esrd.sh stop / esrd.sh start` (or crash) |
| 2 | MCP bridge | WS disconnect → exponential backoff reconnect loop (1s, 2s, 4s, 8s, capped 30s) |
| 3 | new esrd | `Application.start` → `restore_state_from_disk` → re-instantiates feishu-app-session etc |
| 4 | MCP bridge | successful reconnect → `session_register` frame |
| 5 | new ChannelChannel | looks up original peer (may or may not exist after restart) |
| 6a | if PeerServer was restored | rebind `ws_pid`; resume normal operation |
| 6b | if not | reply `{ok: false, error: "session vanished"}` to session_register → MCP decides: retry N times then `session_killed`-itself + exit |

Decision in 6b: after 30s without a PeerServer, MCP exits. Rationale: we don't want orphan MCPs hanging around when the topology is gone.

## 8. `final_gate.sh --live` v2 — 6 artifacts

```
nonce="SMOKE-$(rand)"
tag="smoke-$nonce"

# === setup ===
esrd start --instance=smoke-live         # auto-restores adapters.yaml
esr workspace add esr-dev \
    --cwd ~/Workspace/esr \
    --start-cmd scripts/esr-cc.sh \
    --role dev \
    --chat $FEISHU_TEST_CHAT_ID:$FEISHU_APP_ID:dm

# feishu-app-session auto-instantiated by esrd startup hook
# (if adapters.yaml from previous run had the instance)

# === L1: command spawn ===
L1_id=$(lark_post "/new-session esr-dev tag=$tag")
  → within 30s, Lark server-side chat list shows a bot reply
    "session $tag ready" (contains $nonce via the tag)
  → artifact-1: L1_message_id (the command post)
  → artifact-2: esrd log contains "actor_id=cc:$tag"

# === L2: inbound msg → MCP notification → CC reply ===
lark_post "please reply with: ack-$nonce"
  → within 30s, chat shows bot reply containing "ack-$nonce"
  → artifact-3: the ack reply message_id (via L4-style server query)
  → artifact-4: esrd log contains "tool_invoke reply req_id=..."
    with args.content="ack-$nonce"

# === L5: esr cmd stop → session_killed → MCP exit ===
esr cmd stop feishu-thread-session --param thread_id=$tag --param chat_id=$FEISHU_TEST_CHAT_ID
  → within 10s, esrd log contains "session_killed published session_id=$tag"
  → artifact-5: tmux window $tag no longer exists
    (tmux list-windows check)

# === L6: parallel isolation ===
```

## 10. Risks & mitigations

**R1: MCP stdio protocol version drift**
Claude Code may change its `--dangerously-load-development-channels`
interface; cc-openclaw currently uses a specific shape. Mitigation:
copy cc-openclaw's MCP bootstrapping code (init handshake, tool
schema format) as-is to minimise drift risk.

**R2: SessionRegistry leaks offline rows**
If CC/MCP crashes without clean shutdown, the row stays in :offline
forever. Mitigation: GenServer periodic GC — offline rows older than
24h are purged.

**R3: Reconnect race on esrd restart**
If multiple MCPs reconnect simultaneously when esrd comes back, the
re-instantiation of feishu-app-session might race with the session
rebinds. Mitigation: esrd startup hook completes
`restore_state_from_disk` before Endpoint begins accepting WS (ordering
in Application.start children list).

**R4: Group chat bot identity**
v0.2 requires explicit `(chat_id, app_id)` per chat entry. If operator
gets this wrong in workspaces.yaml, esrd will post as the wrong app or
fail with "bot not in chat." Mitigation: `esr workspace validate`
command that calls Lark `chat.members.list` for each entry and
confirms the declared app is actually present.

**R5: Loopguard LG-6 (scenarios allowlist)**
New scenario file `e2e-esr-channel.yaml` requires updating LG-6's
allowlist sha. Same constraint as §7.3 — single human-synchronised
commit. Document this in the implementation plan so the ralph-loop
doesn't try to "fix" it automatically.

## 11. Contract boundaries

This spec defines **normative** interfaces; implementers choose
internal data structures:

- WS protocol frames (§5.3) — MUST use exact kind names
- MCP tool names + signatures (`reply`, `react`, `send_file`) — MUST
  be JSON-RPC compatible with the shapes in §3.1
- `workspaces.yaml` schema (§5.1) — MUST parse with
  `schema_version: 1`
- Env vars (§5.2 inline) — MUST be present when CC is spawned by
  esr-cc.sh

Non-normative (internal, changeable):
- SessionRegistry ETS layout
- PeerServer handle_info message shapes
- Exact reconnect backoff curve
- `esr-cc.sh` steps ordering

## 12. Open questions (to resolve in implementation planning)

- **Q-A**: Should `esr session chat-id <sid>` be its own CLI or part
  of `esr actors inspect`? Probably fold into inspect output.
- **Q-B**: `workspaces.yaml` — persist to `~/.esrd/default/` or repo-

  future-proofing even though v0.2 only uses `default`.
- **Q-C**: When L6 parallelism exceeds N=5 sessions, does the

  simpler; session is gone.

These questions get closed during `writing-plans` based on concrete

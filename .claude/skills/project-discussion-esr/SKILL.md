## Q&A Flow

Every answer must be backed by evidence — file:line reference **or** test output. Never speculate silently; mark unverified as `[unverified]`.

### Step 0: Freshness check

Before answering, run the following freshness probes (cheap, reliable):

1. `git log --oneline --since="2026-04-21" -- .` — any commits after bootstrap? If yes, the module index may be stale.
2. `ls -lt .artifacts/code-diffs/ .artifacts/e2e-reports/ 2>/dev/null` — any new artifacts since bootstrap? Skim the latest `.md` to understand the change.
3. If the question names a specific file and the file no longer exists at the indexed path, run `bash scripts/refresh-index.sh --module <name>`.

Skip Step 0 if the user's question is purely architectural (no specific file/test reference).

### Step 1: Locate the module

Consult the **Module index** below. If unclear which module owns the concern, use the **User-flow → module map**. For new modules not in the index, run `bash scripts/refresh-index.sh` (full rescan) or spawn a focused subagent.

### Step 2: Read the code

Use the `Read` tool on the current file (paths in `references/module-details.md` are the snapshot at bootstrap time — re-verify if Step 0 flagged changes). Always include `file_path:line_number` in the answer.

### Step 3: Run the test

Run the module's test-runner to get **live** output:

```
bash .claude/skills/project-discussion-esr/scripts/test-<module>.sh
```

The baseline for each module is in `references/module-details.md`; diverge from baseline → explain why (recent code change, flaky test, etc.).

### Step 4: Query prior artifacts

Check `.artifacts/` for context. Prefer `skill-6-artifact-registry` query commands if available; otherwise `ls` + `head`:

```
bash ~/.claude/plugins/cache/ezagent42/dev-loop-skills/0.1.0/skills/skill-6-artifact-registry/scripts/query.sh \
  --project-root /home/yaosh/projects/esr --type eval-doc --status archived
```

Relevant artifact types when triaging:
- `eval-doc` with status `archived` — previously-triaged bug reports with `rejection_reason`
- `e2e-report` — recent test-run outcomes
- `code-diff` — recent PR summaries
- `coverage-matrix` — always at `.artifacts/coverage/coverage-matrix.md`

### Step 5: Compose the answer

Format:
1. Direct answer to the question.
2. Evidence line(s): `file_path:line — quoted behavior` + test-runner output if relevant.
3. If an `eval-doc` in Step 4 is directly relevant, cite it (`eval-doc-007: rejected because …`).
4. Anything unconfirmed → `[unverified]`, not a guess.

### Step 6: Triage outcome (only when discussing an eval-doc or bug report)

If the current discussion *is* about an eval-doc or issue (Phase 7 bug-triage flow):

1. State the conclusion explicitly: **"This is a bug" / "Not a bug" / "Need more info"**.
2. Ask the user to confirm.
3. After confirmation:
   - **Bug** → leave issue open; note "Next: Skill 2 to generate test-plan for this".
   - **Not bug** → run `bash scripts/close-issue.sh --issue-url <url> --reason "<summary>"`, then mark the eval-doc `status: archived` and append `rejection_reason:` + `rejected_at: 2026-04-…` to its frontmatter. Commit.

## Test Pipeline Info (for Skill 3)

When Skill 3 writes pytest code for ESR, it needs:

- **Framework:** pytest 9.0.3 + pytest-asyncio (asyncio_mode = auto); ExUnit for runtime Elixir layer.
- **Python runner:** `uv run pytest` (always; never bare `python -m pytest`). Project enforces via hook: "Direct python/python3 is not allowed. Use uv run instead."
- **E2E scenario runner:** `esr scenario run <name>` (backed by YAML scenario manifest). See `scenarios/e2e-esr-channel.yaml` for the canonical shape.
- **Conftest:** `py/tests/conftest.py` (single root); individual adapter/handler packages have their own `conftest.py` under their `tests/` dir if needed.
- **Import mode:** `importlib` (supports duplicate test-file basenames across adapter/handler packages — see `py/pyproject.toml:46`).
- **testpaths:** `tests`, `../adapters/feishu/tests`, `../adapters/cc_tmux/tests`, `../handlers/{feishu_app,feishu_thread,tmux_proxy,cc_session}/tests`. NOTE: `adapters/cc_mcp/tests` is intentionally NOT included (CC-MCP-1 gap).
- **Naming:** `test_*.py`; pytest auto-discovers. For cross-cutting handler tests use `test_handlers_cross_cutting.py` style.
- **Fixtures pattern:** small inline fixtures in each test file; a few shared live-capture JSON fixtures in `adapters/feishu/tests/fixtures/live-capture/`. No DB fixtures.
- **Evidence collection:** scenarios emit `ledger` events; `scripts/ledger_append.py` writes append-only log. `asciinema` is optional (2.4.0 installed, for pre-release walkthroughs only).
- **Purity rules:** handlers must only import from `{esr, typing, dataclasses, pydantic, enum, __future__, <own_package>}`. Violations caught by `tests/test_handlers_cross_cutting.py` + `esr-lint` CI.
- **Elixir test runner:** `cd runtime && mix test`. Use `--max-cases 1 --seed 0` for reproducible serial runs (avoids CliChannelTest flakes).
- **Scenario YAML shape:** `setup:`, `steps:` (each with `run`, `expect_exit`, `expect_stdout_match`), `teardown:`. Canonical example: `scenarios/e2e-esr-channel.yaml`.
